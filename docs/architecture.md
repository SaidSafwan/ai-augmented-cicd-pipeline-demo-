## Key principle

AI is embedded in the pipeline as a CLI (`AiPipeline`) вҖ” **not** exposed as REST
endpoints. GitHub Actions invokes it at specific stages and gates the build on critical
findings. AI reports are written to the GitHub Actions **Step Summary**.

## Components

- **`Api/`** вҖ” ASP.NET Core (.NET 8) app that gets built and deployed. Exposes
  `/api/health`, Swagger, and a `ProductsController` of **demo-lever endpoints**
  (`/api/products`, `/api/products/{id}` with `?slow`/`?crash`, `POST /api/orders`)
  used to deliberately trigger Azure Monitor alerts on stage. Contains no AI code.
  Application Insights is registered only when a connection string is configured.
- **`AiPipeline/`** вҖ” .NET 8 console CLI. Reads `AZURE_OPENAI_*` from environment
  variables, calls Azure OpenAI (GPT-4.1-mini), and signals pass/fail via exit codes:
  - `review` вҖ” AI code review on the diff вҖ” **gates on Critical/High**
  - `gen-tests` вҖ” suggested xUnit tests as an artifact (advisory)
  - `validate-deploy` вҖ” analyzes the build/deploy log вҖ” **gates on critical risk**
  - `analyze-incident` вҖ” root-cause analysis on pipeline failure (advisory)
- **`.github/workflows/ci.yml`** вҖ” orchestrates the stages above on push to `main`.
